## 1. Контекст и принципы

### Почему Шаг 1 — это фундамент

Все последующие шаги — разметчик эталонов, движок бэктеста, индикаторы, стратегии — полностью зависят от качества данных. Результат бэктеста на некорректных данных бесполезен и опасен: система будет оптимизировать параметры под артефакты, а не под реальные паттерны рынка.

Шаг 1 считается завершённым только когда выполнены три условия:
- Данные собираются непрерывно и система это гарантирует
- Качество данных верифицируется автоматически
- Состояние всей системы сбора видно в одном месте — вкладка Пульс

### Главное правило работы с данными

Обучаемся на старых данных (6–18 месяцев назад).  
Тестируем на средних (3–6 месяцев назад).  
Торгуем на новых (последние 1–3 месяца).

**Никогда не обучаться и тестировать на одних и тех же данных.** Это называется подгонка под историю (overfitting) — индикатор будет идеально работать в прошлом и проваливаться в будущем.

### Классификация данных по восстанавливаемости

Это архитектурный принцип который влияет на приоритет мониторинга и восстановления.

**Восстанавливаемые** — если сборщик упал, данные можно догрузить через REST API. Свечи OHLCV, базовые метрики, Funding Rate, Open Interest (с задержкой).

**Невосстанавливаемые** — если пропустил момент, данных нет нигде. Ликвидации в реальном времени, точный поток сделок для CVD, динамика стакана поминутно. Потеря этих данных — безвозвратна.

Вывод: WebSocket соединения которые собирают невосстанавливаемые данные имеют наивысший приоритет восстановления. Их остановка — критическое событие.

## 2. Архитектура сборщика данных

### Четыре независимых потока

Сборщик состоит из четырёх полностью независимых потоков. Каждый — отдельный модуль с собственным статусом в Пульсе.

**Поток 1 — Спот WebSocket**  
Что собирает: тики сделок, обновления стакана, свечи в реальном времени.  
Тип данных: частично невосстанавливаемые (поток сделок для CVD).  
Приоритет восстановления: высокий.

**Поток 2 — Спот REST**  
Что собирает: исторические свечи OHLCV, синхронизация при запуске, заполнение дыр.  
Тип данных: восстанавливаемые.  
Приоритет восстановления: средний.

**Поток 3 — Фьючерсы WebSocket**  
Что собирает: тики сделок, стакан, ликвидации, агрегированные сделки.  
Тип данных: невосстанавливаемые (ликвидации, точный поток сделок).  
Приоритет восстановления: критический.

**Поток 4 — Фьючерсы REST**  
Что собирает: Open Interest, Funding Rate, Long/Short Ratio, исторические свечи фьючерсов, цена маркировки и индексная цена.  
Тип данных: частично восстанавливаемые.  
Приоритет восстановления: средний.

### Таймфреймы — стратегия агрегации

С биржи получаем только свечи 1m (спот и фьючерс отдельно).  
Все остальные таймфреймы вычисляются локально из 1m:

```
1m (с биржи, спот)
1m (с биржи, фьючерс)
  ├── 3m, 5m, 15m, 30m
  ├── 1h, 2h, 4h, 6h, 12h
  ├── 1d
  ├── 1W (из 1d)
  └── 1M (из 1d, календарный)
```

Агрегация запускается как фоновая задача при каждом закрытии 1m свечи.  
Результат агрегации помечается в БД как `source = 'aggregated'`.

### Rate-Limit Guard — BingX ограничения

Все запросы к BingX проходят через единый Rate-Limit Guard.

Лимиты BingX (актуально на 2026, проверять в документации):
- REST: 20 запросов в секунду на IP
- WebSocket: до 30 подписок на соединение
- Переподключение: не чаще 1 раза в 5 секунд

Три приоритета запросов:
- CRITICAL — исполнение ордеров, данные аккаунта
- HIGH — актуальные рыночные данные для торговли
- NORMAL — исторические данные, синхронизация, верификация

При приближении к лимиту (использовано более 80%) — запросы NORMAL приостанавливаются. При 95% — приостанавливаются HIGH. CRITICAL никогда не приостанавливается.

Статус использования rate-limit отображается в Пульсе в реальном времени.

## 3. Спот и фьючерсы

### Цены разные — всегда

Спотовая и фьючерсная цена никогда не совпадают полностью. Разница называется базис (basis).

Базис = Фьючерсная цена − Спотовая цена

Базис — самостоятельный индикатор настроения рынка:
- Базис положительный и растёт → рынок в контанго, преобладают лонги
- Базис отрицательный → бэквордация, преобладают шорты или страх
- Резкое изменение базиса → часто предшествует сильному движению

### Правило разделения в системе

Стратегия которая торгует фьючерсами работает на фьючерсных данных.  
Стратегия которая торгует спотом работает на спотовых данных.  
Использование спотовых данных для фьючерсной стратегии — только осознанно, как дополнительный контекст, не как основа сигнала.

Все запросы к данным указывают тип рынка явно: `market_type = 'spot' | 'futures'`.  
Это исключает случайное смешение данных разных рынков.

### Уникальные метрики фьючерсов

Следующие метрики физически недоступны на спотовом рынке:

| Метрика | Источник | Восстанавливаемость |
|---|---|---|
| Open Interest | REST + WS | Частично |
| Funding Rate | REST (каждые 8ч) | Да |
| Long/Short Ratio | REST | Частично |
| Ликвидации | WS только | Нет |
| Цена маркировки | REST + WS | Частично |
| Индексная цена | REST + WS | Частично |
| Базис | Расчёт из спот+фьюч | Расчётный |

### Важно для бэктеста

Ликвидации происходят по цене маркировки, а не по рыночной цене. Если не собираешь цену маркировки — невозможно точно моделировать ликвидации в бэктесте. Это искажает результаты стратегий с высоким плечом.

## 4. Невосстанавливаемые данные

### Что теряется при остановке WebSocket

**Ликвидации.** История ликвидаций через публичный REST API BingX практически недоступна. Каждая ликвидация существует только в момент события. Пропустил — потерял навсегда.

**Точный поток сделок для CVD.** Через REST доступны последние N сделок, но исторический поток с точной разбивкой buy/sell — нет. CVD за пропущенный период будет некорректным или отсутствовать.

**Динамика стакана.** Снимок стакана через REST — текущее состояние. История изменений стакана поминутно — только если записывал сам. Нигде не хранится и не отдаётся.

### Последствия потери

Пропуск невосстанавливаемых данных помечается в базе как `data_gap` с типом и длительностью. Периоды с пропусками помечаются как ненадёжные для бэктеста. Бэктестер предупреждает если работает на данных с пропусками невосстанавливаемых метрик.

### Автоматическое восстановление восстанавливаемых данных

При обнаружении дыры в восстанавливаемых данных — автоматически запускается задача догрузки через REST. Это уже реализовано как авторемонт базы свечей. Задача ставится в очередь с приоритетом NORMAL и не мешает основному сбору.

## 5. Верификация данных

### Зачем нужна верификация

Авторемонт исправляет дыры — пропущенные свечи. Но дыра это не единственная проблема. Данные могут присутствовать но быть некорректными — неправильная агрегация таймфрейма, артефакт при записи, рассинхронизация WS. Верификация находит такие случаи.

### Четыре уровня проверки

**Уровень 1 — Полнота.** Все ли минуты присутствуют за проверяемый период. Нет ли дыр в последовательности временных меток.

**Уровень 2 — Точность OHLCV.** Запрашиваем тот же диапазон через REST и сравниваем значения. Допустимое отклонение — менее 0.01% (погрешность округления).

**Уровень 3 — Корректность агрегации.** Запрашиваем готовый таймфрейм с биржи (например 1h) и сравниваем с нашей агрегацией из 1m. Совпадение должно быть точным. Это проверяет правильность алгоритма агрегации.

**Уровень 4 — Непрерывность.** Нет ли аномальных скачков между соседними свечами которые могут быть артефактом сбора. Порог — изменение цены более чем на 3× ATR за одну свечу без подтверждения объёмом.

### Статусы верификации

Каждый диапазон данных (символ + таймфрейм + период) имеет один из статусов:

- `unverified` — ещё не проверялся
- `in_progress` — проверяется прямо сейчас
- `verified` — проверен, расхождений нет
- `verified_partial` — проверен частично (не все уровни)
- `mismatch_found` — обнаружены расхождения
- `repaired` — расхождения были, исправлены автоматически
- `needs_review` — расхождение слишком большое для авторемонта, требует ручного рассмотрения

### Доверительный рейтинг данных

Каждый диапазон получает числовой рейтинг доверия от 0 до 100. Складывается из:
- Сколько раз проверен
- Когда последний раз проверен
- Были ли расхождения и насколько серьёзные
- Насколько критичен период для активных стратегий

Бэктестер видит этот рейтинг. Результат бэктеста на данных с рейтингом ниже 70 помечается как ненадёжный.

### Умное расписание проверок

Не статичное "проверять каждые N часов" а приоритизация по риску:

**Высокий приоритет** — данные которые будут использоваться для торговли в ближайшие 24 часа. Проверяются первыми и чаще.

**Средний приоритет** — данные для активных бэктестов. Проверяются в фоне.

**Низкий приоритет** — данные которые уже проверялись и не изменились. Интервал проверки увеличивается экспоненциально — проверен вчера → следующая через 2 дня → через 4 → через 8.

**Триггерная проверка** — если детектор аномалий замечает странное значение, автоматически запускается точечная проверка этого периода вне расписания.

### Связь с авторемонтом

Верификация обнаруживает и помечает расхождения.  
Авторемонт принимает решение что делать.

Если расхождение в пределах порога → исправить автоматически, записать в лог.  
Если расхождение выше порога → пометить `needs_review`, уведомить через Пульс, не исправлять без подтверждения.

Пороговые значения — настраиваемые параметры, не захардкоженные.

## 6. Политика хранения и сжатия

### Оценка роста базы

Реалистичная оценка при активном сборе всех данных (5 пар, спот + фьючерс):
- Свечи 1m: ~250 МБ в год
- Поток сделок: ~1–2 ГБ в месяц
- Снимки стакана (каждые 10 сек): ~1–2 ГБ в месяц
- Снимки рынка, метрики: ~200 МБ в месяц

Итого: 2–5 ГБ в месяц, 25–60 ГБ в год.

### Три температуры данных

**Горячие — последние 90 дней**  
Основная база, быстрый доступ. Полное разрешение по всем типам данных. Используются для торговли и верификации.

**Тёплые — от 90 дней до 12 месяцев**  
Отдельная схема или файл. Агрегированные данные стакана и потока сделок. Свечи в полном разрешении. Используются для бэктеста.

**Холодные — старше 12 месяцев**  
Сжатый архив. Загружается только по явному запросу для долгосрочного бэктеста. Только агрегированные данные.

### Правила агрегации по типу данных

| Тип данных | До 1 месяца | 1–3 месяца | 3–12 месяцев | Старше года |
|---|---|---|---|---|
| Свечи 1m | Полностью | Полностью | Полностью | Агрег. до 5m |
| Поток сделок | Полностью | Агрег. в CVD/1m | Агрег. в CVD/5m | Удалить |
| Снимки стакана | Каждые 10 сек | Каждые 5 мин | Каждый час | Удалить |
| Снимки рынка | Каждые 10 мин | Каждый час | Каждый день | Каждый день |
| Ликвидации | Полностью | Полностью | Агрег./час | Агрег./день |
| OI / Funding | Полностью | Полностью | Полностью | Полностью |

### Умная очистка

Данные которые прошли верификацию (статус `verified`) и использовались в бэктесте — помечаются флагом `used_in_backtest`. При агрегации и очистке таких данных можно действовать смелее — их корректность подтверждена.

### Архив важных периодов

Периоды с подтверждёнными разворотами высокого качества (класс A и B из разметчика эталонов — Шаг 2) экспортируются в отдельный файл и хранятся постоянно. Это обучающий датасет который только ценнее со временем. Размер небольшой — только отмеченные события.

### Мониторинг роста в Пульсе

В Пульсе отображается:
- Текущий размер базы (горячая / тёплая / холодная)
- Скорость роста за последние 7 дней
- Прогноз когда закончится свободное место
- Статус и время последней агрегации
- Следующая запланированная агрегация

### Сжатие файлов

SQLite поддерживает сжатие через расширения. Исторические данные (тёплые и холодные) сжимаются с коэффициентом 3–5× за счёт повторяющихся паттернов. Горячие данные не сжимаются — приоритет скорость доступа.

## 7. Вкладка Пульс

### Концепция

Пульс — единственное место где виден статус всей системы. Отвечает на три вопроса одним взглядом: всё работает, данные чистые, есть ли что-то требующее внимания.

Вкладка расширяемая — в будущем добавляются новые блоки без изменения существующих.

### Блок 1 — Соединения (всегда виден, не скроллится)

Постоянная панель в верхней части. Каждое соединение — один индикатор со статусом и временем последнего успешного ответа.

Соединения для отображения:
- WS Спот BingX (по каждой активной паре)
- WS Фьючерсы BingX (по каждой активной паре)
- REST API BingX (спот)
- REST API BingX (фьючерсы)
- Fear & Greed API
- Новостной фид
- Локальная БД (горячая)
- WebSocket сервер UI

Дополнительно: использование rate-limit в реальном времени — полоса прогресса, текущие запросы / лимит в минуту. При приближении к 80% — жёлтый, при 95% — красный.

### Блок 2 — Состояние модулей

Каждый модуль системы — одна строка. Пять столбцов: название, статус, время последнего действия, счётчик событий за минуту, текущая задержка.

Градация статусов:
- Работает нормально (зелёный)
- Работает медленно — задержка выше нормы (жёлтый)
- Деградирует — отклонения нарастают (оранжевый)
- Завис — нет активности дольше порога (красный мигающий)
- Остановлен намеренно (серый)

### Блок 3 — Очередь задач

Три секции: активные задачи (с прогрессом для долгих), ожидающие в очереди (с приоритетом и временем ожидания), последние завершённые (время выполнения, статус).

Примеры периодических задач которые здесь видны:

| Задача | Триггер | Приоритет |
|---|---|---|
| Пересчёт агрегированных ТФ | Каждая закрытая 1m свеча | P1 |
| Обновление базиса (спот-фьючерс) | Каждую минуту | P1 |
| Верификация данных (выборочно) | По расписанию + триггер | P2 |
| Агрегация старых данных | Раз в сутки, ночью | P2 |
| Обновление Fear & Greed | Каждый час | P1 |
| Обновление OI и Funding Rate | Каждые 5 минут | P1 |
| Снимок состояния рынка | Каждые 10 минут | P1 |
| Авторемонт базы свечей | Триггер: обнаружена дыра | P1 |
| Точечная верификация | Триггер: аномалия в данных | P1 |
| Сжатие тёплых данных | Раз в неделю | P2 |

### Блок 4 — Лента критических событий

Только WARNING, ERROR, CRITICAL. Не очищается автоматически. Каждое событие можно отметить как "увидено". Группируется по модулю. Показывается время появления и длительность если проблема продолжается.

### Блок 5 — Состояние данных

Таблица по парам: последняя свеча получена, наличие дыр за последние 24 часа, статус верификации последнего периода, доверительный рейтинг, размер занимаемого места.

### Блок 6 — Поток событий (умный)

Общий поток но с группировкой. Однотипные события от одного модуля за последние 60 секунд схлопываются в одну строку с счётчиком. Раскрываются по клику.

Аномалии и ошибки никогда не схлопываются — каждая остаётся отдельной строкой.

Фильтры: по модулю, по типу события, по паре, по уровню важности. По умолчанию скрыты рутинные события (закрытие свечей, плановые пересчёты).

## 8. Очередь задач

### Три приоритета — три разных механизма исполнения

**P0 — Critical (< 50ms)**  
Один выделенный asyncio worker. Ничто не может его прервать.  
Задачи: исполнение ордеров, реакция на flash crash, срабатывание SL/TP, алёрт spoofing.

**P1 — Background (< 500ms)**  
Пул из трёх async workers. Работают параллельно.  
Задачи: пересчёт индикаторов, агрегация таймфреймов, верификация, обновление метрик.

**P2 — Batch (без ограничений по времени)**  
ProcessPoolExecutor, отдельные процессы. Не блокируют event loop.  
Задачи: бэктест, ML-оптимизация, разметка эталонов, сжатие данных.

### Почему ProcessPoolExecutor для Batch

Бэктест и оптимизация — это тяжёлые вычисления с numpy и pandas. Из-за GIL Python они не могут реально параллелиться в потоках. Отдельный процесс — единственный способ использовать несколько ядер CPU не блокируя основной event loop и не замедляя торговлю.

### Видимость в Пульсе

Все задачи из всех трёх очередей видны в Блоке 3 вкладки Пульс. Для каждой задачи: название, приоритет, статус, время в очереди, время выполнения, результат.

## 9. Watchdog и эскалация

### Проблема которую решаем

WebSocket останавливается без предупреждения — нигде не отображается, помогает только перезапуск сервера. Это недопустимо для критических потоков невосстанавливаемых данных.

### Три уровня самодиагностики для каждого WS

**Уровень 1 — Ping/Pong.**  
BingX требует heartbeat каждые 30 секунд. Если pong не пришёл в ответ на ping в течение 5 секунд — фиксируем как деградацию, пробуем ещё раз.

**Уровень 2 — Счётчик сообщений.**  
Если за последние 5 секунд не пришло ни одного сообщения по активной подписке — деградация. Не обязательно WS упал — возможно биржа задержала данные.

**Уровень 3 — Контрольная сумма.**  
Периодически сравниваем последнюю цену из WS с последней ценой из REST. Расхождение выше порога — рассинхронизация, деградация.

Только если все три уровня показывают проблему — это смерть модуля.

### Четыре стадии эскалации

**Стадия 1 — Норма.**  
Зелёный индикатор в Пульсе. Тихий лог.

**Стадия 2 — Деградация.**  
Жёлтый индикатор. Запись в ленту критических событий с временем начала. Попытка мягкого восстановления (переподключение без сброса состояния).

**Стадия 3 — Потеря связи.**  
Красный индикатор. Алёрт в Пульсе. Автоматический запуск серии переподключений с экспоненциальным backoff. Прогресс переподключения виден в UI.

**Стадия 4 — Мёртв.**  
Все попытки исчерпаны. Мигающий красный. Звуковой сигнал если приложение открыто. Торговля автоматически переводится в режим Alert-only. Требует ручного вмешательства.

### Правило для невосстанавливаемых данных

Если WS фьючерсов (ликвидации) перешёл в Стадию 2 — немедленно фиксируем временную метку начала деградации. Период от этой метки до восстановления помечается в БД как `data_gap` с типом `liquidations_unavailable`. Бэктестер учитывает эти пометки.

## 10. Изменения в схеме БД

### Обновление таблицы candles

Добавить поле `market_type TEXT NOT NULL DEFAULT 'spot'` — значения `'spot'` или `'futures'`.  
Добавить поле `data_trust_score INTEGER DEFAULT 100` — доверительный рейтинг 0–100.  
Уникальный индекс должен включать market_type: `UNIQUE(symbol, timeframe, open_time, market_type)`.

### Новая таблица: futures_metrics

Хранит метрики специфичные для фьючерсного рынка. Open Interest, Funding Rate, Long/Short Ratio, Mark Price, Index Price, Basis — всё с временными метками и символом.

### Новая таблица: liquidations

Хранит каждое событие ликвидации: символ, время, сторона (long/short), цена, объём, тип (принудительная/авто). Только WebSocket источник, не восстанавливаемо.

### Новая таблица: data_verification_log

Хранит результаты каждой верификации: символ, таймфрейм, период начала/конца, уровень проверки, статус, процент совпадения, количество расхождений, исправлено или нет, время проверки.

### Новая таблица: data_gaps

Хранит все известные пропуски данных: символ, тип данных, время начала/конца пропуска, причина, восстановимость, статус восстановления.

### Обновление таблицы market_snapshots

Добавить поле `market_type` для разделения спотовых и фьючерсных снимков.  
Добавить поле `basis` — значение базиса на момент снимка.

### Новая таблица: storage_stats

Хранит ежедневную статистику объёма базы по температурам: горячая/тёплая/холодная, размер в МБ, количество записей, скорость роста. Используется Пульсом для прогноза заполнения диска.
